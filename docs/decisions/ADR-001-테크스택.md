# ADR-001: 테크스택

## Status

Accepted

## Date

2026-07-11

## Context

초기 저장소는 frontend와 backend를 함께 개발하며, 개발 환경과 운영 환경의 데이터베이스 동작 차이를 줄여야 한다. 개발자는 애플리케이션을 `pnpm`으로 핫리로드하면서 데이터베이스만 Docker 컨테이너로 실행한다. Node.js의 현재 활성 LTS를 기준으로 런타임을 고정하고, 워크스페이스·웹·API·ORM·데이터베이스의 책임을 명확히 할 필요가 있다.

## Decision

패키지 관리는 `pnpm workspaces`만 사용하고 `engines`에 Node.js 24 LTS를 명시한다. 저장소는 `apps/frontend`의 Next.js와 `apps/backend`의 NestJS로 구성한다. backend의 데이터 접근에는 Prisma ORM을, 개발과 운영을 포함한 모든 환경의 관계형 데이터베이스에는 PostgreSQL을 사용한다.

`pnpm`은 엄격한 의존성 관리와 효율적인 디스크 사용을 제공하며 workspaces만으로 현재 모노레포 요구를 충족한다. Prisma는 타입 안전한 클라이언트, 스키마 기반 migrate, Prisma Studio, 활발한 생태계 모멘텀을 제공한다. PostgreSQL을 전 환경에 사용하면 Docker로 개발 DB를 초기화할 수 있어 인메모리 대역이 필요 없고 SQL 방언 차이도 제거된다. Node.js 24는 활성 LTS이므로 보안·유지보수 지원 기간을 확보한다.

## Alternatives considered

### npm/yarn workspaces

- Pros: 널리 알려진 기본 도구이며 도입 장벽이 낮다.
- Cons: 의존성 격리와 디스크 효율에서 `pnpm`의 일관된 운영 모델을 제공하지 못한다.
- **Rejected:** 현재 요구에는 `pnpm workspaces` 단독 사용이 더 단순하고 재현 가능한 의존성 관리를 제공한다.

### TypeORM

- Pros: NestJS 생태계에서 널리 사용되며 데코레이터 기반 모델링을 지원한다.
- Cons: 타입 안전한 질의 경험과 스키마·마이그레이션·관리 도구의 일관성이 Prisma보다 약하다.
- **Rejected:** Prisma의 타입 안전성, migrate, Studio 및 생태계 모멘텀이 초기 팀의 생산성에 더 적합하다.

### 인메모리 개발 데이터베이스

- Pros: 별도 DB 컨테이너 없이 빠르게 시작할 수 있다.
- Cons: PostgreSQL과의 SQL 기능 및 방언 차이로 운영 전 결함을 숨길 수 있다.
- **Rejected:** Docker DB 초기화로 개발 편의성을 확보하면서 전 환경 PostgreSQL parity를 유지할 수 있다.

## Consequences

### Enables

- frontend와 backend의 독립 배포 단위를 하나의 workspace에서 관리한다.
- 개발·운영 간 데이터베이스 동작 차이를 줄이고 스키마 변경을 migrate로 추적한다.
- DTO와 ORM 모델 사이의 타입 검증 기반을 마련한다.

### Costs / trade-offs

- 개발자는 Node.js 24와 `pnpm`을 설치해야 하며 PostgreSQL 컨테이너를 실행해야 한다.
- Prisma 스키마와 생성된 클라이언트의 갱신 절차를 지켜야 한다.

### New constraints

- 모든 패키지는 Node.js 24 LTS engines 선언을 준수한다.
- 앱은 호스트에서 `pnpm`으로 실행하고 개발 DB만 Docker 컨테이너로 실행한다.
- 운영과 개발 모두 PostgreSQL을 사용한다.
